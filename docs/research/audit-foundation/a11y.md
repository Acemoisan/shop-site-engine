# Accessibility (a11y) — Audit Knowledge Base

WCAG 2.x AA guidance for Calgary local-shop sites, synthesized from the a11y claim set. _Every claim in this dimension is confidence "provisional"; all points below should be read with that caveat._

## Necessary (must / should-have)

**The automatable "top six" the engine must eliminate by construction.** The WebAIM Million 2026 found ~96% of automatically-detectable failures fall into six categories, present on ~95.9% of home pages: low-contrast text (~84%), missing image alt (~53%), missing form labels (~51%), empty links (~46%), empty buttons (~31%), and missing document `lang` (~13.5%). Guaranteeing these six at build time removes the large majority of failures a real shop site would otherwise ship. _(provisional)_

Must-haves that map directly to this engine:
- **Color contrast (1.4.3 / 1.4.11)** is the single most consequential automatable criterion — most frequent failure AND fully verifiable. Treat AA contrast (≥4.5:1 normal text, ≥3:1 large/non-text) as a hard, enforced constraint on the OKLCH token pairs, not per-shop judgment. Re-validate for every new `theme.css`. _(provisional)_
- **Image alt (1.1.1)** — content/photographic images (gallery, service cards) must require alt text; use `alt=""` only for decorative images. Enforce alt as a required field in `ShopContent` and as an editable Storyblok field for any client-uploaded content image. _(provisional)_
- **Form labels (1.3.1 / 3.3.2 / 4.1.2)** — every contact-form control needs a persistent visible label programmatically associated via `<label for>`/id or `aria-label`; placeholder-only fields fail. _(provisional)_
- **Accessible names for links/buttons (2.4.4 / 4.1.2)** — every icon-only link/button (`tel:`, map pin, social, nav booking icon, mobile toggle) must carry an accessible name. `Icon.astro` renders bare SVG, so any interactive element containing only an Icon is an empty link/button without an `aria-label`. _(provisional)_
- **Document language (3.1.1)** — emit `<html lang="en">` (or correct locale) in the shared base layout. Cheapest of the six to fix permanently. _(provisional)_
- **Error identification (3.3.1)** — on form validation failure, identify the field and describe the error in text (not color-only), tied via `aria-describedby` + `role=alert`. Native HTML5 constraint validation satisfies the baseline. _(provisional)_
- **Target size (2.5.8, AA)** — interactive targets ≥24×24 CSS px or adequately spaced; ~44px recommended for primary actions. _(provisional)_
- **Focus visible (2.4.7) + keyboard (2.1.1)** — every control must show a visible focus indicator; do not strip the default outline without a replacement. Ship a token-based `:focus-visible` style in `base.css`. _(provisional)_
- **Reflow (1.4.10) + resize text (1.4.4)** — single-column responsive stacking and rem/Tailwind type scale largely satisfy 320px reflow and 200% text zoom; confirm in the screenshot stage. _(provisional)_
- **Hero text over photos (1.4.3)** — overlaid hero text must contrast against the actual photo pixels; a fixed scrim does not guarantee this when a client uploads a bright image. Flagged as the highest-leverage a11y fix in the engine. _(provisional)_

Should-haves: heading structure (single H1, no skipped levels), `prefers-reduced-motion` guard on hover/scale animations (2.3.3 is AAA, so best-practice), `autocomplete` tokens on personal-data fields (1.3.5, also a conversion win), valid `lang` value, and avoiding text-as-image (1.4.4/1.4.5).

## Niche / situational

- **Skip-to-content link (2.4.1, Level A)** — nice-to-have for these short single-page sites; trivial to bake once into the shared layout. _(provisional)_
- **Heading-order / single-h1 check** — axe tags this best-practice, NOT a WCAG success criterion; ship as advisory with higher false-positive tolerance. _(provisional)_
- **`meta-viewport` zoom-not-disabled (1.4.4, AA)** — refines the existing `mobileViewport` inventory item to reject `user-scalable=no` / `maximum-scale<2`; the engine's own templates pass, but prospect-audit targets often fail. _(provisional)_
- **`html-lang-valid` / `valid-lang`** — low impact; a permissive BCP-47 regex keeps false positives near zero (en, en-CA, fr, fr-CA cover Calgary). _(provisional)_
- **Federal/Ontario law edge cases** — the Accessible Canada Act applies only to federally regulated orgs (banks, telecom, transport), and AODA applies only in Ontario; neither binds a typical Calgary shop. A federally regulated client (credit union, airline-affiliated service) would be the rare exception. _(provisional)_

## Always reused vs rare

**Always reused (own centrally, never per-shop):**
- Token-pair contrast validation — every shop swaps the OKLCH palette, so contrast is the #1 a11y regression in a token-themed engine. _(provisional)_
- `<html lang>`, a shared `:focus-visible` ring, a global `prefers-reduced-motion` reset block, and a robust default hero scrim — each is "written once, inherited by every shop." _(provisional)_
- Alt-text as a required CMS content-model field for client-uploaded images. _(provisional)_
- The Tier-1/Tier-2 split for `packages/audit`: Tier-1 static checks (lang, title, viewport-zoom, image-alt, label, button-name, link-name, heading-order) extend the existing `inventoryFromHtml` pattern; Tier-2 (color-contrast) reuses the Playwright harness. _(provisional)_

**Rare:**
- The 7-item manual pass (keyboard walkthrough, focus visibility, focus order, one screen-reader pass, alt-quality review, 200%/reflow check, form label/error check) — required to honestly claim AA, but only run per finished site. _(provisional)_
- A build-time token contrast gate via `culori` (`wcagContrast`/`wcagLuminance` on parsed `oklch()` strings) — a single script, rarely changed once written. _(provisional)_
- `document-title` and APCA-as-advisory checks — low frequency, included for completeness. _(provisional)_

## Most vs least common

**Most common failures (highest real-world prevalence):**
1. Low-contrast text — ~84% of pages (83.9% in 2026; 79.1% in 2025), ~34 instances/page. _(provisional)_
2. Missing image alt — ~53% of pages; ~18.5% of all images lack alt, ~45% of those are linked images (also producing non-descriptive links). _(provisional)_
3. Missing form input labels — ~51% of pages. _(provisional)_
4. Empty links — ~46%. _(provisional)_
5. Empty buttons — ~31%. _(provisional)_

**Least common / lower-frequency:** missing document language (~13.5%), missing `<title>` (rare as a failure but free to check), and the manual-only AA criteria (focus visibility, tap-target sizing) that automated scans cannot measure at all, so they never appear in prevalence stats despite being real AA requirements. _(provisional)_

## What works vs what doesn't

**What works:**
- Automated scans are reliable on a small, high-value set: contrast, name/role/value, and language account for ~58.8% of real defects by volume (axe-core catches ~57.38% of all issues in Deque's study), even though only ~13% of WCAG 2.2 AA criteria (7 of 55) are fully automatable. Lean on automation where it's strong (especially contrast). _(provisional)_
- The engine already does several things right: `Hero.astro` correctly uses `alt=""` on its decorative background; `Faq.astro` uses native `<details>/<summary>` (accessible, avoids empty-button risk); `ContactNAP.astro` and `CTA.astro` use visible phone text (passes name/context); shared components use semantic `<a>` and don't strip focus outlines; real text (not images of text) is rendered for menus/prices/FAQs. _(provisional)_
- A token contrast gate can be fully automated at build time with no browser. WCAG contrast is hue-independent, so one sRGB-luminance check per token pairing covers all color-vision types — no per-deficiency passes needed. _(provisional)_

**What doesn't work / known gaps:**
- **Lighthouse/PageSpeed score is NOT proof of AA.** Automated engines fully evaluate only ~30-40% of failures (criteria-count view: ~13% reliable, ~45% partial, ~42% undetectable). A site can score 100 and still be severely inaccessible. W3C WCAG-EM: "Absence of detected errors does not indicate that a page is accessible." Relabel the a11y grade as "automated-detectable issues," never "AA conformance," off a score alone. _(provisional)_
- **OKLCH L is not a WCAG substitute.** Contrast must be computed from sRGB relative luminance, not OKLCH lightness; at equal L, higher chroma shifts the ratio, so a "safe-looking" L gap can still fail. Gate on the real sRGB ratio. APCA (WCAG 3) is informational only — report it, but gate on WCAG 2.x. _(provisional)_
- **Engine-specific defects flagged:**
  - `Hero.astro` overlays `text-white` on a *fixed* dark gradient regardless of the client photo — a bright/snow image can drop effective contrast below 4.5:1. Bake in a strong default scrim (≈60% black) or a contained text panel. (Highest-leverage fix.) _(provisional)_
  - `SiteNav.astro` hides section links on mobile (`hidden ... sm:inline`) with no hamburger menu — mobile users lose in-page nav (severity lowered only because they are on-page anchors reachable by scroll). _(provisional)_
  - Themes often set `--border` very light (e.g. barber `--border` L=0.88 vs `--background` L=0.98, well under 3:1) — fails 1.4.11 wherever the border is the sole affordance for a form field. _(provisional)_
  - `Hours.astro` uses `<ul>` flex rows of two `<span>`s — day→hours pairing is visual-only; prefer `<dl>`/`<dt>`/`<dd>` or a table with `scope`. Low real-world severity. _(provisional)_
  - No `prefers-reduced-motion` guard on the Hero's `hover:scale` transitions; no skip link. _(provisional)_

**Per-vertical notes:**
- **Law, dental** — higher regulatory/ADA-adjacent exposure; most sensitive to overclaimed "AA compliant" language. Scope claims to build-time best-practice. _(provisional)_
- **All verticals (barber, cafe, spa, dental, auto, retail)** — provincially regulated, so outside the federal ACA. Alberta has no in-force private-sector web-accessibility statute (Bill 206 did not pass as of June 2026); the real exposure is the Alberta Human Rights Act s.4 duty-to-accommodate (reactive, complaint-driven, undue-hardship defence — "plausible," not settled, exposure). US-style demand-letter litigation is an ADA Title III phenomenon and does NOT transfer to Canada — do not import that fear framing. _(provisional)_

## Audit takeaways

- **Score the automatable top six explicitly** (contrast, alt presence, form labels, link/button names, document lang) as a graded a11y dimension — near-universal failure among prospects makes a clean build a real outreach differentiator. _(provisional)_
- **Build the audit in two tiers:** Tier-1 deterministic static checks (lang, lang-valid, title, viewport-zoom, image-alt, label, button-name, link-name, heading-order) over fetched HTML using a DOM parser (cheerio/linkedom, not regex) to cut false positives; Tier-2 color-contrast via the existing Playwright harness on the built preview. _(provisional)_
- **Separate "fail" from "needs-review/incomplete"** — contrast over background-images/gradients/opacity, alt *quality* (presence ≠ accuracy), and placeholder-as-label are the main false-positive sources; report presence-only honestly. _(provisional)_
- **Add a build-time token contrast gate** (`culori`) asserting the six required pairings: `--foreground/--background`, `--muted-foreground/--muted` and `/--background`, `--primary-foreground/--primary`, `--accent-foreground/--accent` (all ≥4.5:1), and `--border/--background` (≥3:1 where border is a functional boundary). Allow-list logo/wordmark and disabled-state text; don't flag solid-button background-vs-page contrast. _(provisional)_
- **Never claim "WCAG AA compliant" from a score.** Use build-time, effort-based language: e.g. "Built to meet WCAG 2.2 AA best practices at handoff — semantic HTML, AA-contrast tokens, keyboard navigation, alt-text slots, accessible forms," with a disclaimer that the client owns post-handoff content changes. Position the a11y grade as risk-reduction + reach, not legal compliance. _(provisional)_
- **Gate the highest-leverage engine fixes centrally:** a strong default hero scrim, a shared token-based `:focus-visible` ring, a global `prefers-reduced-motion` reset, required CMS alt fields, and `aria-label` by default on any icon-only interactive element. _(provisional)_
