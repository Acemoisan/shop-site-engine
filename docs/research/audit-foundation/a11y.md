# Accessibility (WCAG 2.x AA) — Audit Briefing

What a Calgary local-shop site engine must guarantee, what an audit can actually detect, and how to talk about it honestly — synthesized from the re-verified a11y claim base.

## Necessary (must / should-have)

The single highest-leverage facts first, then the rest of the build-time guarantees.

**The "WebAIM Million" top six account for ~96% of all auto-detectable WCAG failures.** Low-contrast text, missing image alt, missing form labels, empty links, empty buttons, and missing document language lead the list five-plus years running. 95.9% of one million home pages failed at least one (56.1 errors/page avg). An engine that guarantees these six by construction eliminates the large majority of failures a real shop site would otherwise ship — these are necessary but not sufficient for full AA.

The six, with web-wide prevalence and how the engine kills each by construction:

- **Low-contrast text (~84% of pages)** — the #1 failure and the single most consequential automatable criterion. AA floor is 4.5:1 normal / 3:1 large (≥24px or ≥18.66px-bold) / 3:1 non-text. Because each shop swaps the OKLCH palette in `theme.css`, contrast must be re-validated per theme, never assumed inherited — it is the highest-frequency a11y regression in a token-themed engine.
- **Missing image alt (~53%)** — the #2 failure. Content/photographic images (hero, gallery, service photos) require descriptive alt; decorative images use `alt=""`. Make `alt` a required field on every non-decorative image slot in `ShopContent` and on any Storyblok-uploaded content image, so the build cannot ship a content image without it.
- **Missing form labels (~51%)** — every contact/search/newsletter control needs a programmatic `<label for>`/id (or `aria-label`); never placeholder-only. The contact form is the highest-risk surface and ships on essentially every site.
- **Empty links (~46%)** — icon-only links (social, click-to-call, map, nav booking glyph) need an accessible name. Inline-SVG `Icon.astro` inside an `<a>` is the specific at-risk pattern; require `aria-label` on every icon-only interactive element by convention.
- **Empty buttons (~31%)** — icon-only buttons (mobile menu toggle, icon submit) need text or `aria-label`. Lower risk here since most CTAs are text links, and `Faq.astro` uses native `<details>/<summary>`.
- **Missing document language (~13.5%)** — the cheapest of the six: one `<html lang="en">` in the shared base layout covers every shop forever. Assert it on built HTML.

**Conversion-surface guarantees (the small, fixed interactive set every shop ships):**

- **`tel:` phone links** must convey a call action; visible phone-number text passes, but an icon-only call link must carry `aria-label="Call (xxx) xxx-xxxx"` (SC 4.1.2 + 1.1.1).
- **CTA text + background contrast** — `--primary`/`--primary-foreground` and `--accent`/`--foreground` pairs on buttons must hit 4.5:1 (3:1 for large).
- **Keyboard reach + visible focus** — every CTA/link must be tabbable with a perceivable focus style. Shared components use semantic `<a>` (good); the real risk is a global reset killing `:focus-visible` (SC 2.1.1 + 2.4.7).
- **Hero text over photography** must meet 4.5:1/3:1 against the *actual pixels*. `Hero.astro` overlays `text-white` on a fixed gradient scrim — a bright client-uploaded photo can drop effective contrast below 4.5:1. Bake in a strong-enough default scrim (or a contained text panel) so contrast holds for any photo. This is the highest-leverage single a11y fix in the engine.
- **Contact-form fields** need a persistent visible label, programmatically associated (placeholder-only fails: 3.3.2 + 1.3.1); correct `autocomplete` tokens (1.3.5, also a conversion win); and on error, an in-text message identifying the field, tied via `aria-describedby`/`role=alert` (3.3.1) — native HTML5 constraint validation satisfies the baseline.
- **Target size** — interactive targets ≥24×24 CSS px or adequately spaced (SC 2.5.8 AA). Shared button padding (`px-6 py-3`, nav `px-4 py-2`) clears it; set a `min-height/width` default to be safe.
- **Mobile nav** must keep all destinations operable; if a hamburger toggle is added it must be keyboard-operable with focus state and ≥24px target.
- **Reflow / resize** — single-column responsive stacking satisfies 1.4.10 reflow (320px / 400%) and 1.4.4 200% text zoom; mostly free from Tailwind, just verify the Hero's clamped display type and fixed-height bands.

**Token-system contrast model (must validate per theme):**

- WCAG contrast is sRGB relative-luminance math, **not** OKLCH lightness — tokens must be converted to sRGB and run through the luminance formula; OKLCH L is only a rough heuristic _(provisional)_.
- Pairings to validate: `--foreground/--background`, `--foreground`/`--muted` and `--muted-foreground`/`--muted`, `--muted-foreground/--background`, `--primary-foreground/--primary`, `--accent-foreground/--accent` (all ≥4.5), and `--border/--background` ≥3 only where the border is a functional boundary. `--muted-foreground` is the highest-risk token.
- This gate **can be fully automated at build time** with `culori` (`wcagContrast`/`wcagLuminance` parse `oklch()` directly) — read `theme.css`, build the pairing matrix, fail the build below threshold. No browser needed.

**Honesty guarantees (must-have, framing):**

- A Lighthouse/PageSpeed a11y score (even 100) is **not** AA conformance — engines fully evaluate only ~30–40% of criteria. Relabel the grade "automated-detectable issues," never claim "AA" off a score alone.
- W3C's WCAG-EM is authoritative: "Absence of detected errors does not indicate that a page is accessible." A defensible AA claim requires automated scan **plus** manual evaluation.
- Don't claim "AA compliant"/"fully accessible" for a no-retainer build: accessibility is point-in-time and degrades as the client self-edits in Storyblok. Defensible claim: *"Built to meet WCAG 2.2 AA best practices at handoff — semantic HTML, AA-contrast tokens, keyboard nav, alt-text slots, accessible forms."*

## Niche / situational

- **`<input type=image>` / `<area>` / `<object>` alt, `frame-title`** — sibling Level-A static checks for rarer markup; include for completeness.
- **`meta-viewport` zoom-not-disabled** (1.4.4 AA) — refine the existing `mobileViewport` check to also reject `user-scalable=no`/`maximum-scale<2`. Near-zero FP; our templates pass, but prospect targets often fail.
- **`document-title`** (2.4.2 A) — overlaps the SEO grade; free to add.
- **Non-underlined links** — must clear 4.5:1 vs background **and** 3:1 vs surrounding body text (1.4.1); underlining links sidesteps the 3:1-vs-text leg entirely (simplest default).
- **Hours block** (`Hours.astro`) — current `<ul>` of flex rows conveys day→hours only visually; a `<dl>` or `<table scope=row>` makes the relationship programmatic (1.3.1, low real severity since DOM order keeps them adjacent).
- **Map/directions link** — address-as-link-text passes; an icon-only map pin needs `aria-label`.
- **`prefers-reduced-motion`** — gate hover/scale transitions (best-practice AAA, hence should-have); one global block in `base.css` covers every shop.
- **Skip-to-content link** (2.4.1 A) — modest benefit on a short single-page site; trivial to bake once. Nice-to-have.
- **Heading order / single-h1** — axe tags this **best-practice, not WCAG** — flag as advisory, not a hard AA fail (~15–25% FP).
- **Disabled/logo/decorative text** is exempt from contrast — allow-list these (e.g. the nav wordmark) to avoid false failures.
- **APCA** (WCAG 3) is informational only — gate on the WCAG 2.x ratio, report APCA as advisory.

## Always reused vs rare

**Always reused (bake once into the shared engine, inherited by every shop):**
- `<html lang>` and `<title>` in the base layout.
- A `:focus-visible` ring in `base.css` on a token, so no shop can ship invisible focus.
- The `prefers-reduced-motion` global guard.
- A strong default hero scrim so overlay contrast holds for any uploaded photo.
- Required `alt` field in the content model / Storyblok for content images; `alt=""` for decorative backgrounds (a CMS-content-model guarantee, not just a component one).
- Per-theme token contrast gate (the one thing that varies per shop and silently breaks AA).

**Rare / situational:** the manual minimum check set (run once per build, ~15–30 min); `meta-viewport` zoom and `document-title` (cheap completeness adds); `culori` token gate (one script, run at build).

## Most vs least common

**Most common real-world failures (web-wide):** low-contrast text (~84%) > missing alt (~53%) > missing form labels (~51%) > empty links (~46%) > empty buttons (~31%) > missing lang (~13.5%). Contrast plus name/role/value plus language account for the bulk of actual defects — a small check set captures most wins.

**Least common / lowest-impact:** missing `document-title`, heading-order skips, BCP-47 lang-validity, and skip-link absence — all low-prevalence or low-severity, included mostly for completeness.

**Coverage reality (why "most common" ≠ "most checks"):** by criteria count only ~13% of WCAG 2.2 AA (7 of 55) are reliably auto-flagged and ~42% can't be automated at all; by issue *volume* axe-core catches ~57% (a few high-frequency types, mainly contrast, dominate). A GDS audit found the best single tool hits ~30–40%, 2–3 tools combined ~50%. Cite ~57% as the issue-volume upper bound and ~13–30% by criteria — never round up to imply conformance.

## What works vs what doesn't

**Engine patterns that already pass (keep them):**
- `Hero.astro` sets `alt=""` on its decorative background — correct.
- `Faq.astro` uses native `<details>/<summary>` — avoids empty-button risk and is keyboard-accessible.
- Shared components use semantic `<a>` and don't remove the default focus outline.
- `ContactNAP.astro` / `CTA.astro` use visible phone-number text and address-as-link-text — both pass.
- Real text (not images of text) in `Services.astro`/`Faq.astro` — satisfies 1.4.4/1.4.5; risk is only client-uploaded graphics containing menu/price text.
- Tailwind responsive single-column layout largely satisfies reflow and 200% zoom for free.

**Engine gaps to fix:**
- `Hero.astro` fixed gradient scrim can under-contrast over bright client photos.
- `SiteNav.astro` renders section links `hidden ... sm:inline` with no hamburger — mobile users lose in-page nav (severity lowered because on-page anchors are still reachable by scrolling).
- Hover/scale transitions ship with no `prefers-reduced-motion` guard.
- Light `--border` tokens (e.g. barber `--border` L=0.88 vs `--background` L=0.98) fail 1.4.11 wherever the border is the sole form-field affordance — a real, automatable defect.
- `Hours.astro` day→hours pairing is visual-only.
- No skip link.

**Per-vertical notes:**
- **Law, dental** — higher regulatory/ADA-adjacent exposure, so the "automated + manual pass" claim language (not "AA compliant") matters most here; applies to all verticals but these are most sensitive.
- **All verticals** — an icon-only call/booking glyph added to a vertical's nav must get `aria-label` by default; the current text-based CTAs pass.
- A solid `--primary` button with a contrasting label does **not** need its background to clear 3:1 vs the page (avoid over-flagging); only bordered/outline controls where `--border` is the sole boundary need the 3:1 check.

## Audit takeaways

- **Score the six, gate the contrast.** Make `packages/audit` assert the WebAIM top six on built HTML (lang, title, image-alt presence, label, button-name, link-name, viewport-zoom — Tier 1, static/DOM-parser), and run a `culori` build-time gate on the `theme.css` token pairings (the per-shop AA regression). Use cheerio/linkedom over regex for element-enumeration checks to cut false positives _(provisional, on the Tier-1/Tier-2 split)_.
- **Contrast needs a browser; the rest don't.** Color-contrast on rendered pages requires computed styles (reuse the existing Playwright harness or inject axe-core), and ambiguous cases (text over images/gradients/opacity) must surface as *needs-review/incomplete*, never as fails.
- **Report presence, not quality.** Static checks verify alt/label *presence*, not accuracy — label findings as "N of M images missing alt," and pair them with the 7-item manual set (keyboard walkthrough, focus visibility, focus order, one screen-reader pass, alt-text quality, 200%/reflow, form labels) before any AA claim.
- **Fix the engine once.** Bake in: hero scrim, `:focus-visible` ring, `prefers-reduced-motion` guard, `<html lang>`/`<title>`, required content-image `alt`, and per-theme contrast validation — these convert near-universal failures into guarantees every shop inherits.
- **Sell quality and risk-reduction, not legal compliance.** Alberta has no in-force web-accessibility statute (Bill 206 did not pass); the ACA and AODA don't reach a provincially regulated Calgary shop; the real exposure is the Alberta Human Rights Act duty-to-accommodate (reactive, complaint-driven, *plausible* not established for websites). US demand-letter litigation does **not** transfer to Canada — avoid fear-selling. Disability is the dominant human-rights complaint ground nationally _(provisional)_, so the duty is real but rarely exercised.
- **Differentiator is genuine.** ~95% of prospect sites fail at least one detectable criterion, so a build that clears the automatable subset is a real outreach edge — graded as "reduces human-rights-complaint exposure and improves reach," never "makes you legally compliant."
